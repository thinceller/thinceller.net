# Introduce Playwright for E2E Testing

## Status

accepted

## Context

プロジェクトにはユニットテストやE2Eテストが存在せず、以下の課題があった：

- **ライブラリアップデートの不安**: 依存関係更新時の影響範囲が不明確で、手動での動作確認に依存
- **デプロイ前の確認負荷**: 主要ページ、RSS、ナビゲーションなどを手動で確認する必要
- **リグレッション検出**: 既存機能の破壊を事前に検出する仕組みがない
- **CI/CDの品質担保不足**: 自動テストなしでマージ・デプロイが行われる

特に個人開発では、リファクタリングや依存関係更新のたびに全機能を手動確認するのは現実的でない。

## Decision

**Playwright** をE2Eテストフレームワークとして導入する。

### テスト構成

- **対象ブラウザ**: Chromium（デスクトップChrome）
- **テストポート**: 3001（開発サーバーの3000と分離）
- **テストスイート**:
  - スモークテスト: 主要ページの表示確認（/, /about, /blog, /blog/[slug]）
  - RSSフィード検証: /blog/rss.xml, /blog/atom.xmlの妥当性確認
  - ナビゲーションテスト: ページ間遷移の動作確認
- **CI統合**: GitHub Actionsでプルリクエスト時に自動実行

### Considered Options

**1. Playwright（採用）**
- Microsoft製、高速、複数ブラウザ対応
- TypeScript統合、Auto-waiting機能
- GitHub Actions公式サポート、効率的なキャッシング

**2. Cypress**
- 人気が高い、豊富なエコシステム
- ただしパフォーマンスでPlaywrightに劣る

**3. テストなし**
- 手動確認に依存し続ける
- リグレッションリスクが残る

### 選択理由

- **パフォーマンス**: 並列実行とブラウザ起動が高速
- **開発者体験**: TypeScript統合、豊富なアサーション、デバッグUI
- **CI統合**: GitHub Actions公式サポート、キャッシング効率が良い
- **将来の拡張性**: 複数ブラウザ対応が容易（現在はChromiumのみ）

## Consequences

**より容易になったこと:**
- CIでE2Eテストを実行することで、動作確認の手間が削減された
- 安心してライブラリの更新などを行えるようになった
- リグレッションの早期発見が可能になった
- デプロイ前の信頼性が向上（本番環境相当のビルドでテスト実行）

**より困難になったこと:**
- CI実行時間が増加（ビルド + テスト実行で数分追加）
- テストの保守コスト（UIや機能変更時にテストも更新が必要）
- フレイキーテスト対策が必要（Playwrightの自動待機とリトライで緩和）
- 依存関係の増加（`@playwright/test` とChromiumバイナリ約300MB、CIではキャッシング）
