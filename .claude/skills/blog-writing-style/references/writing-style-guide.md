# 執筆スタイルガイド

このドキュメントは、筆者のブログ記事（`_posts/*.mdx`）の執筆スタイルを定義している。
過去の全記事（9記事、2021年〜2026年）の分析に基づいて策定されたものである。

---

## 文体・トーン

### 基本文体

- **「です・ます」調**で統一する
- 技術的な解説であっても堅くなりすぎず、親しみやすいトーンを保つ
- 読者に対して対等な立場で語りかけるスタンスを取る

### 一人称

- **「私」を基本**とする
- 内省的な文脈や自身の経験を語る際に**「自分」も許容**する
  - OK: 「自分にとっては〜」「自分が好きな〜」「自分でも使い倒しながら」
  - OK: 「私が普段の業務で〜」「私はこれまで〜」「私もMaxプランを〜」
- 同一記事内で「私」と「自分」が混在しても問題ない
- ただし、同一段落内で不自然に交互に使用するのは避ける

### 控えめな主張スタイル

断定的な表現を避け、控えめに主張するのが筆者の特徴である。以下のような表現を好む:

- 「〜だと感じています」
- 「〜と思いました」
- 「〜のように感じます」
- 「〜かもしれません」
- 「〜そうです」「〜ようです」
- 「〜だと考えています」
- 「〜気がします」

**避けるべき表現**:
- 「〜に違いありません」（強い断定）
- 「〜すべきです」（指示的・上から目線）
- 「〜は間違っています」（否定的断定）
- 「〜してください」（読者への指示。手順説明を除く）

### 口語的なニュアンス

適度にくだけた表現を挟むことで親しみやすさを出す。以下は実際に使用されている表現の例:

- 「〜なーと思った」
- 「まずいかな」
- 「好みなのは」
- 「ですかね」
- 「ともあれ」
- 「要は」
- 「〜した気がします」
- 「〜というくらい」

全体のトーンを崩さない程度に使用する。1記事あたり数箇所が適切で、過度に使用すると文体の品質が下がる。

### 率直な自己開示

弱みや課題を隠さず素直に書くのが筆者の重要な特徴である:

- 「コンプレックスになっており」
- 「付け焼き刃なのですが」
- 「危機感を抱くように」
- 「人に見られる前提の文章を書くことを避けがちな自分を矯正するため」
- 「業務で使った経験がほぼない（中略）ため微妙なところです」

この率直さは筆者の個性であり、レビューで削除・弱化すべきではない。

### 語尾のバリエーション

同じ語尾が3回以上連続するのは避ける。以下のように語尾を変化させる:

- 「〜ました」「〜です」「〜ています」を交互に使う
- 体言止めを適度に挟む
- 括弧書きで補足する形を使う

## 記事構成パターン

すべての記事に共通する骨格:

- **導入**: 背景・動機を1-2段落で述べる（descriptionの内容と対応させる）
- **本文**: 記事タイプに応じた構成
- **まとめ**: 総括 + 今後の展望への言及（単なる要約で終わらない）

以下、記事タイプ別の構成パターンを定義する。

### パターンA: 技術移行・導入記事

技術やサービスの移行・導入を報告する記事。

**構成**:
1. **導入**: 移行/導入の概要と背景（1-2段落）
2. **背景/動機**: なぜ移行/導入したか（動機の丁寧な説明）
3. **技術構成/前提**: 現状のシステム構成の説明
4. **移行/導入手順**: ステップバイステップの説明
5. **詰まった点**（ある場合）: 遭遇した問題と解決策
6. **まとめ/所感**: 感想 + 今後の展望

**補足**: 詰まった点がなかった場合は省略してよい。その場合、手順の説明を充実させるか、所感を厚めにする。

**該当記事例**: VercelからCloudflareに移行した、Next.js 15から16へのアップグレード

### パターンB: ツール・プロダクト作成記事

自作ツールやプロダクトの紹介記事。

**構成**:
1. **導入**: 作ったものの概要と動機（過去記事への参照を含むことが多い）
2. **技術的背景**: 関連する技術やコンセプトの説明
3. **技術選定/設計方針**: なぜその技術を選んだか
4. **機能紹介/使い方**: 主要な機能のデモや説明
5. **まとめ**: 感想 + 今後の展望

**該当記事例**: git-ccをRustで作った、ブログを作った

### パターンC: 学習記録記事

技術学習の経過を記録する記事。

**構成**:
1. **導入**: 何を学んでいるかの概要
2. **現状の説明**: 自身のスキルセットや背景
3. **動機**: なぜ学ぼうと思ったか（複数の観点から丁寧に記述する）
4. **手応え/学んだこと**: 具体的な学習内容、使った教材、気づき
5. **今後**: 次のステップ、展望

**該当記事例**: 最近 Rust を勉強している

### パターンD: 問題解決記事

特定の技術的問題の発見から解決までを記録する記事。

**構成**:
1. **導入/問題の発見**: 何が起きたかの説明
2. **問題の詳細**: 具体的な挙動や再現方法
3. **解決策の探索**: 試した解決策（不完全なものも含めて過程を記録する）
4. **最終的な解決策**: 採用した解決方法の詳細
5. **まとめ**: 学びの総括

**特徴**: 状況が変化した場合、記事の冒頭に追記を設けることがある。

**該当記事例**: Chakra UIのLinkコンポーネントとfocus-visible疑似クラス

### パターンE: アップデート報告記事

ライブラリやツールのアップデートによる変化を報告する記事。

**構成**:
1. **導入**: アップデートの概要と過去記事への参照
2. **Before**: アップデート前の状況（数値データを含む）
3. **After**: アップデート後の状況（数値データを含む）
4. **変更内容の説明**: 技術的な変更点の解説
5. **まとめ**: 関連動向や今後の展望への言及

**該当記事例**: @opennextjs/cloudflare 1.2.0 でアップロードサイズが大幅に減った

### パターンF: 振り返り・エッセイ記事

年末の振り返りなど、エッセイ的な記事。

**構成**:
1. **導入**: 振り返りの経緯や背景（なぜブログで書くかなど）
2. **各トピック**: カテゴリ別に見出しを分けて記述（仕事、技術、プライベートなど）
3. **今後の目標**: 来年の目標や展望
4. **まとめ**: 全体の総括

**該当記事例**: 2025年の振り返り

### 構成に関する共通ルール

- **導入で動機を丁寧に説明する**: 「なぜやったか」は「何をやったか」と同等以上に重要。導入が薄い記事は構成として不十分。
- **まとめで展望に触れる**: 単なる要約ではなく、今後の展望や関連する動向への言及で締めくくる。
- **過去記事への参照**: 関連する過去記事がある場合は積極的にリンクする。形式: `[記事タイトル](ファイル名)`
- **見出し階層**: h2 → h3の順で使用する。h1は記事タイトルに相当するため使わない。
- **descriptionとの対応**: 導入部分はフロントマターのdescriptionと対応する内容にする。

## コンテンツ品質

### コード例

- **必要最小限**にとどめる。長大なコードブロックは避ける
- 要点を伝えるのに必要な分だけ掲載する
- コードブロックには言語を指定する（例: ```tsx, ```bash）
- ファイル名がある場合は `言語:ファイル名` の形式で指定する（例: ```tsx:MyLink.tsx）

### OgpCardの使用

- 外部リンク（GitHub、公式ドキュメント、参考記事など）はOgpCardコンポーネントで表示する
- 形式: `<OgpCard url="..." />`
- 記事本文での説明は要点に留め、詳細はリンク先に委ねる
- 連続して複数のOgpCardを配置してもよい

### 過去記事への参照

- 関連する過去記事がある場合はMarkdownリンクで参照する
- 形式: `[記事タイトル](ファイル名)`
- 例: `[最近 Rust を勉強している](2022-01-30-learning-rust)`
- 「〜に書いたように」「〜で記載した通り」のようなフレーズで自然に組み込む

### フロントマター

必須項目:
```yaml
---
title: "記事のタイトル"
description: "記事の概要説明"
publishedTime: "YYYY-MM-DDTHH:mm:ssZ"
---
```

オプション:
- `modifiedTime`: 更新日時（ISO8601形式）
- `tags`: タグ配列

descriptionは記事の内容を1-3文で要約する。導入の内容と対応させる。

## 用語・表記の一貫性

### 表記ルール

同一記事内では表記を統一する:

- ライブラリ名・サービス名は公式表記に従う（例: Next.js, Cloudflare, Chakra UI, Rust, TypeScript）
- カタカナ語の長音: 一般的な表記に従う（コンパイラ、サーバー、ユーザー）
- 漢字・ひらがなの使い分け: 読みやすさを優先する
  - 「出来る」→「できる」（ひらがな推奨）
  - 「事」→「こと」（ひらがな推奨）
  - 「為」→「ため」（ひらがな推奨）

### 筆者が好む定型表現

以下は筆者が頻繁に使用する表現パターン:

- 「〜してみることにしました」（新しいことに挑戦する際）
- 「〜を試してみました」
- 「〜することができました」
- 「〜していきたいです」（展望を述べる際）
- 「〜は参考になります」（外部リソースの紹介時）
- 「〜の利点だと感じました」
- 「〜が大きな経験になったと思います」
- 「〜する所存です」（記事の締めに使用されることがある）
